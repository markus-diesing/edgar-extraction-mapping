1. System Overview

Architecture Summary

A local, standalone pipeline for processing SEC EDGAR 424B2 filings. The backend is a Python FastAPI application; the frontend is a React/Vite single-page app. All persistent state lives in a SQLite database and the local file system. No cloud infrastructure, no authentication, no Docker required.

2. Installation & Setup

Prerequisites

  • macOS (tested on Sonoma/Sequoia, Apple Silicon and Intel)
  • Python 3.11 or higher (python3 --version)
  • Node.js 18 or higher (node --version)
  • npm 9+ (comes with Node)
  • An Anthropic API key (for classify + extract)

Backend Setup

# 1. Navigate to backend directory
cd /path/to/EDGAR-Extraction_Mapping/backend

# 2. Create and activate virtual environment
python3 -m venv .venv
source .venv/bin/activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Set the Anthropic API key (required for classify/extract)
export ANTHROPIC_API_KEY="sk-ant-..."

# 5. Initialise the database (creates tables, safe to re-run)
python main.py init-db

Frontend Setup

cd /path/to/EDGAR-Extraction_Mapping/frontend
npm install

Running the App

Two terminals are required:

# Terminal 1 — Backend
cd /path/to/EDGAR-Extraction_Mapping/backend
source .venv/bin/activate
export ANTHROPIC_API_KEY="sk-ant-..."
uvicorn main:app --reload --port 8000
# Terminal 2 — Frontend
cd /path/to/EDGAR-Extraction_Mapping/frontend
npm run dev

Open http://localhost:5173 in your browser.

Optional: persist API key across sessions Add export ANTHROPIC_API_KEY="sk-ant-..." to your ~/.zshrc or use python scripts/setup_key.py to store the key in macOS Keychain.

Migration System

The database._migrate() function runs on every startup. It applies a list of ALTER TABLE ... ADD COLUMN statements idempotently (SQLite raises an OperationalError if the column already exists, which is silently ignored). To add a new column:

  1. Add the column to the ORM model class in database.py.
  2. Add an entry to the migrations list in _migrate().
  3. Restart the backend — the column is added automatically on existing databases.

6. Extraction Pipeline Deep Dive

Tier 0 — Registry Extraction

Entry point: backend/extract/html_extractor.py :: extract_registry_fields(filing_record)

Populates fields that are already known from the database filing record — no document parsing required. Currently extracts:

  • identifiers.cusip — from filings.cusip column (confidence = 1.0)
  • parties.issuer.name — from filings.issuer_name column (confidence = 1.0)

The CUSIP value itself comes from the EDGAR filing index (CIK entity lookup), so it has full certainty. The CUSIP mapping spreadsheet (CUSIP_PRISM_Mapping.xlsx) is a separate lookup used only for classification hints, not for field extraction.

Tier 1 — HTML Table Extraction

Entry points: backend/extract/html_extractor.py :: extract_from_html(html, issuer_hints, label_map)

Finding the Key Terms Table

_find_key_terms_table(soup, issuer_table_heading) searches for the two-column Key Terms HTML table using a priority list of headings:

  1. Issuer-specific heading from issuer_hints.key_terms_table_heading
  2. Cross-issuer fallback headings: "key terms and assumptions", "key terms", "terms of the notes", etc.

For each heading candidate, it checks: (a) tables whose first row/header cell contains the heading, (b) heading-like elements (h1–h4, div, p, span) followed by or near a table.

Label Resolution Pipeline

For each label:value pair extracted from the table rows:

  1. label_mapper.resolve_label(raw_label, label_map) — normalise (lowercase, trim, strip colon/footnotes) and look up in the merged label map.
  2. If resolved: call field_parsers.get_parser(field_path) to get the typed parser function.
  3. If a parser exists: call parser(raw_value) to get a typed Python value.
  4. If parser returns None (unparseable): skip the field (LLM will handle it).
  5. If no parser exists: store raw text as-is.
  6. Unresolved labels: add to the LabelMiss list for miss-log persistence.

Label Map Merge Order

Lower priority → higher priority (later wins on conflict):

  1. files/label_map_cross_issuer.yaml — baseline hand-curated entries
  2. files/label_map_user.yaml — user-added entries via the UI
  3. Per-issuer table_labels from the issuer's YAML hints — highest priority

Field Parsers (field_parsers.py)

The FIELD_PARSERS dict maps PRISM field paths to typed parser functions. Array paths use [*] wildcard notation (resolved by label_mapper via prefix matching).

Tier 2 — LLM Extraction

Entry point: backend/extract/extractor.py :: extract_filing(filing_id)

Prompt Construction

The prompt is split into two parts to maximise prompt cache reuse:

  • Preamble (static per model + issuer) — schema JSON, hints block, output format instructions. Sent with cache_control: {type: ephemeral}.
  • Filing text (unique per filing) — stripped plain text, up to MAX_FILING_CHARS (120K chars). Not cached.

The system prompt is also cached. This means only the filing text block is charged at the full input rate; the schema + hints + system prompt benefit from the 0.10× cache read rate after the first call for a given model/issuer combination.

Tool-Call Mode

Extraction uses Anthropic's tool_use feature with a single tool submit_prism_extraction whose input schema wraps the PRISM model schema. Claude is forced to call this tool exactly once (tool_choice: {type: "tool", name: "submit_prism_extraction"}), which guarantees valid structured JSON output — no regex cleanup needed.

The tool input has three top-level keys:

  • prism_data — the populated PRISM object
  • _confidence — {dot.path: float} per-field confidence scores
  • _excerpts — {dot.path: string} source text fragments (max 200 chars)

Key Terms Section Anchoring

Before sending to Claude, the stripped filing text is trimmed using _trim_to_key_terms_section(filing_text, issuer_hints). This locates the Key Terms section using issuer-specific headings first, then cross-issuer fallbacks, and returns a window of MAX_FILING_CHARS characters starting 200 chars before the anchor. This ensures Claude sees the most relevant section even for very long filings.

Merge Logic

The merge in extract_filing() builds a pre_extracted dict from Tier 0 and Tier 1 results. For each PRISM field path:

  • If the path is in pre_extracted → use the Tier 0/1 value (with its higher confidence)
  • Otherwise → use the LLM value from the flattened prism_data
  • Tier 0 → Tier 1 conflict: Tier 1 wins (it parsed from the document; Tier 0 is registry)
  • Any conflict between Tier 1 and LLM is logged at INFO level

The final source tag ("registry", "html_table", or "llm") is stored in field_results.source and shown as badges in the UI.

8. PRISM Schema Management

Schema File Location and Format

Active schema: schemas/prism/prism-v1.schema.json
The schema is a JSON Schema document with a top-level oneOf array, one entry per payout type. Each entry has a properties.model.const that is the payout_type_id, and a $defs section with shared type definitions.

Fetch → Diff → Activate Workflow

  1. POST /api/admin/schema/fetch — downloads the schema from PRISM_SCHEMA_URL, saves to schemas/prism/pending/{timestamp}.schema.json, and computes a diff using schema_diff.py. Returns the diff JSON.
  2. Review the diff in the SchemaManager UI. Each change is classified as breaking, caution, or safe.
  3. POST /api/admin/schema/pending/{fetch_id}/activate — archives the current schema to schemas/prism/archive/, copies the pending schema to the active path, calls schema_loader.reload(), and cleans up pending files.

What Each Change Severity Means for Code

SeverityExample ChangeImpact on Code
BreakingField removed; type changed from string to numberExisting field_results rows may have invalid types. label_mapper entries for removed fields become orphans. Re-extraction needed.
CautionNew enum value added; required field made optionalNo immediate breakage, but field parsers or hints may need updating to handle new values.
SafeNew optional field added; description updatedNo impact on existing data. New field will appear as not_found in previous extractions.

schema_loader.py Internals

backend/schema_loader.py loads and caches the schema at startup. Key functions:

  • list_models() — returns list of payout_type_id strings from schema oneOf
  • get_model_schema(model_name) — returns resolved schema dict for one model (with $refs inlined)
  • get_field_descriptors(model_name) — returns all leaf field paths with types, enum values, required flags
  • load_cusip_mapping() — reads CUSIP_PRISM_Mapping.xlsx, returns {CUSIP: NamedTuple(payout_type_id)}
  • reload() — clears the in-memory cache; called after schema activation
  • get_schema_version() — returns the schema version string from $id

9. Adding a New Issuer

To improve Tier 1 (HTML table) extraction coverage for a new issuer, create a YAML hints file. No code changes are required.

  1. Find a sample filing from the issuer. Ingest one or two filings from the issuer. Open the raw HTML (View Filing in the UI) and identify the Key Terms table.
  2. Create the issuer YAML file. Create files/hints/issuer_IssueName.yaml (e.g., issuer_UBS.yaml). Copy the structure from an existing issuer YAML as a starting point.
  3. Set aliases.name_match_patterns. Add all name variants that appear in EDGAR's issuer_name field for this issuer. The extractor does a case-insensitive substring match against the filing's issuer_name.
  4. Set key_terms_table_heading. Find the exact heading text of the Key Terms table in the sample filing HTML (case-insensitive match is applied).
  5. Add table_labels. For each row in the Key Terms table, add an entry mapping the issuer's label text to the correct PRISM field path.
    Tip: run an extraction first, check the Miss Log in the Label Map tab — it shows exactly which labels appeared but couldn't be mapped.
  6. Add section_headings. These are used to anchor the LLM text window on the relevant section of the filing.
  7. Test with a sample filing. Re-extract the sample filings. Check how many fields now show the TABLE badge vs AI. Iterate on the table_labels until coverage is satisfactory.
No server restart needed The hints loader is mtime-cached. Creating or editing a YAML file in files/hints/ is picked up automatically on the next extraction call.

11. Development Guide

Adding a New Field Parser

Field parsers live in backend/extract/field_parsers.py. To add a parser for a new PRISM field path:

  1. Define a new function parse_myfield(raw: str) -> SomeType | None. Return None if the input is unparseable — the caller falls back to the LLM result.
  2. Add the function to the FIELD_PARSERS dict at the bottom of the file, mapping the full PRISM dot-path to the function.
  3. For array fields (e.g., underlyings), use the [*] wildcard notation; label_mapper.get_parser() does prefix matching.

Adding a New PRISM Model

When PRISM adds a new payout type, no Python code changes are needed — just configuration:

  1. Activate the new schema via the Schema Manager UI (fetches from PRISM server).
  2. The new model ID appears in the GET /classify/models response and in the classifier's model list automatically.
  3. Optionally add a section YAML in files/sections/ if sectioned extraction should cover the new fields.
  4. Add cross-issuer field-level hints for any new fields that have special parsing requirements.
  5. Add label map entries for common table labels that map to the new fields.

Frontend Component Patterns

All API calls go through frontend/src/api.js. To add a new API call:

  1. Add a new method to the api object in api.js following the existing pattern (fetch + error handling).
  2. Components use useState + useEffect to load data on mount and useCallback for stable reload handlers.
  3. The main view switch is in App.jsx — add a new view by extending the mainView state and the nav button array.
