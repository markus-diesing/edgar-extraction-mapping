EDGAR Extraction & PRISM Mapping — User Manual

Architecture Tech Handbook
Home / Introduction

1. Introduction

The EDGAR Extraction & PRISM Mapping tool is an internal LPA application that automates the process of pulling structured product data from SEC EDGAR filings and transforming it into the PRISM data format. It runs entirely on your Mac — no internet connection is required beyond the initial filing download and Claude API calls.

What Does It Do?

Think of it as a five-step pipeline:

  1. Ingest. You search EDGAR for a 424B2 filing by issuer name or CUSIP, or paste an accession number directly. The tool downloads the HTML prospectus and stores it locally.
  2. Classify. Claude AI reads the first few pages of the filing and assigns it to a PRISM payout type (e.g., "Barrier Reverse Convertible", "Autocall Note"). This tells the system which fields to extract next.
  3. Extract. The system extracts all PRISM schema fields using three complementary methods: registry metadata (CUSIP, issuer name), HTML table parsing (Key Terms tables), and Claude AI for everything else. Each field is tagged with its source and a confidence score.
  4. Review. You review the extracted fields in the browser, accept or correct values, and approve the filing when satisfied.
  5. Export. The approved data is exported to JSON and CSV files ready for downstream PRISM ingestion.

The EDGAR → PRISM Concept

SEC EDGAR contains thousands of 424B2 filings — the prospectus supplements that structured product issuers file when they issue a new note. These filings contain the full economic and legal terms of the product. PRISM is LPA's internal data model for structured products, with a well-defined JSON schema covering all the key parameters (barriers, coupons, autocall terms, underlying assets, etc.).

The challenge is that every issuer (Barclays, JPMorgan, Citigroup, BofA, UBS, etc.) formats their 424B2 filings differently. This tool bridges the gap by combining deterministic HTML parsing (for the Key Terms table that all issuers include) with AI extraction (for fields not in the table or in issuer-specific formats).

Internal POC — Not for Production Use This is a local test and demonstration system. All data stays on your Mac. No login, no cloud storage, no sharing — just you, the browser, and a local Python server.

2. Getting Started

Opening the App

The app runs as two local services. Both must be running:

  • Backend — FastAPI Python server on http://localhost:8000
  • Frontend — Vite React dev server on http://localhost:5173

Open your browser at http://localhost:5173.

Check the status indicator In the top-right of the header you will see a coloured dot and "Backend OK" or "Backend down". If it shows red/down, the Python backend is not running — check your terminal. If it shows "API key missing", the ANTHROPIC_API_KEY environment variable is not set and classify/extract will fail.

3. Filings View

The Filings view is the main working area. It has a collapsible left sidebar (with the filings list and ingest panel) and a large content area on the right showing the selected filing's detail.

Filing Lifecycle & Status Badges

Each filing moves through a defined sequence of statuses. The current status is always shown as a coloured badge on the filing card and in the detail header.

Status BadgeMeaningNext Action
ingested Filing HTML downloaded and stored. No AI processing yet. Click Classify
classified AI assigned a PRISM payout type with confidence ≥ 75%. Click Extract
needs review Classification confidence was below 75% or result was "unknown". Human review required. Use Set Model to override, then Extract
extracted All PRISM fields extracted. Ready for human review. Review fields, then Approve
approved Human reviewer accepted the extraction. Ready to export. Click Export
exported JSON + CSV files generated in data/exports/. Done — files ready for downstream PRISM ingestion
Resetting a filing The Reset ↺ button (visible on classified/needs_review filings) reverts the filing to "ingested" status, clearing all classification data. This is blocked once extraction has run — you must re-extract or unapprove first.

Sidebar — Filings List & Ingest Panel

Filings List Tab

The filings list shows all filings in the local database, newest first. You can:

  • Search — the search box filters by CUSIP, issuer name, or accession number in real time.
  • Filter by status — use the status dropdown to show only filings at a specific stage.
  • Select a filing — click any filing card to open its detail in the main area.
  • Refresh — the ↻ button reloads the list from the backend.

Ingest Panel Tab

Use this panel to bring new filings into the system. There are two methods:

Method 1 — EDGAR Search

Fill in one or more of the search fields (search term, entity name, CUSIP, date range) and click Search EDGAR. The tool queries EDGAR's full-text search API and returns matching 424B2 filings. Click Ingest on any result to download it.

EDGAR search uses the EFTS (EDGAR Full-Text Search) API. Results are limited to 424B2 form type. You can combine a keyword (e.g., "barrier reverse convertible") with an entity name (e.g., "Barclays") to narrow down results.

Method 2 — Direct Accession Number

If you already know the accession number (format: 0001234567-24-123456), paste it into the "Accession Number" field and click Ingest Filing. This skips the search and downloads the specific filing directly.

EDGAR rate limits EDGAR enforces a rate limit of 10 requests/second per IP. The tool already adds a 200ms delay between requests. For bulk ingestion, prefer off-peak hours (21:00–06:00 US Eastern time).

Filing Detail

Clicking a filing in the sidebar opens its detail view. The detail panel shows:

Header Information

  • Status badge — current lifecycle status
  • CUSIP — 9-character product identifier from EDGAR registry
  • Issuer — the structured product issuer (e.g., "Barclays Bank PLC")
  • Filing date — the date the 424B2 was filed with the SEC
  • Accession number — unique EDGAR identifier
  • PRISM model — the classified payout type (visible after classification)
  • Classification confidence — Claude's confidence in the payout type assignment
  • Product title — exact product name extracted from the cover page

KPI Strip

Below the header, a row of coloured chips shows timing and token metrics for each completed pipeline step:

  • Ingest (blue) — time to download and save the filing HTML
  • Classify (violet) — wall-clock time, token counts, and USD cost of the classification call(s). If stage 2 was triggered, the call count shows "2 calls".
  • Extract (green) — time, tokens, and cost of the extraction. In sectioned mode, shows the number of sections processed.

Classification Override (Set Model)

When the AI classifier produces a wrong or uncertain result, you can use Set Model to manually assign the correct PRISM payout type. A dropdown appears with all valid PRISM model IDs from the current schema. Select the correct type and confirm.

The override is recorded in the classification_feedback table, which feeds back into future classification prompts as few-shot examples — helping the AI learn from corrections.

Simple View — FieldTable

After extraction, the default view shows a table of all extracted PRISM fields. Each row has:

  • Field name — the dot-path PRISM field identifier (e.g., barrier.triggerDetails.triggerLevel)
  • Source badge — how the value was obtained:
    • TABLE — parsed from the Key Terms HTML table (highest trust, confidence 0.97)
    • REG — sourced from EDGAR registry metadata (certainty 1.0 — CUSIP, issuer)
    • AI — extracted by Claude LLM (confidence varies)
  • Confidence bar — coloured bar and percentage. Fields below 80% are highlighted in amber as low-confidence.
  • Value — the extracted value (or a "Not found" indicator)
  • Review status — pending / accepted / corrected / rejected
  • Edit & Accept/Reject buttons — click to review the field inline

Reviewing a Field Inline

  1. Click the Edit icon on a field row to open an inline editor.
  2. Modify the value if needed. The source excerpt (text from the filing that the AI used) is shown for reference.
  3. Click Accept to confirm the value (sets review_status to "accepted").
  4. Click Correct after editing to save the changed value (sets review_status to "corrected").
  5. Click Reject to mark the field as rejected (value won't appear in export).

Expert Review (Side-by-Side)

Toggle the Expert View button to switch to a split-screen layout: the field table on the left and the actual 424B2 HTML filing on the right, displayed in an iframe.

  • Selecting a field — click any field row in the left pane. If Claude provided a source excerpt, the corresponding text is highlighted in the filing HTML on the right using scroll-and-highlight injection.
  • Resizable divider — drag the vertical divider between the two panes left or right to redistribute screen space. The split is clamped between 15% and 85%.
  • Filing viewer — the right pane renders the actual 424B2 HTML (with inline styles preserved). All images downloaded during ingest are served correctly. You can scroll through the full document while a field is selected.
When to use Expert Review Expert Review is most useful when verifying AI-extracted fields (AI badge) or correcting uncertain values. The source highlight makes it easy to see exactly what text Claude used and whether the interpretation was correct.
Home / Expert Settings

4. Expert Settings

Click Expert ⚙ in the top navigation to enter the Expert Settings view. This area contains five tabs for tuning the extraction system. Changes here affect all future extractions — no server restart is needed.

Tab: Field Hints

Field Hints control the guidance given to Claude during extraction. There are three levels, applied in order from most general to most specific.

Cross-Issuer Hints

The top panel shows cross-issuer field-level rules from files/hints/cross_issuer_field_hints.yaml. These rules are injected into every extraction prompt regardless of issuer. Examples:

  • Synonyms for barrier.triggerDetails.triggerLevel (e.g., "Downside Threshold", "Knock-In Level")
  • Format rules for coupon.rate (always output as decimal, e.g. 0.0875 not 8.75%)
  • Cautions for fields that look similar across product types

The cross-issuer hints file is a YAML file. You can edit it here in the UI and save. Changes are picked up immediately on the next extraction call.

Per-Issuer Hints

The issuer selector dropdown lists all issuers that have a YAML hint file in files/hints/ (e.g., issuer_Barclays.yaml, issuer_JPMorgan.yaml). Select an issuer to view and edit its hints.

Key fields in an issuer hint file:

FieldPurpose
aliases.name_match_patternsList of strings used to match the issuer name from EDGAR registry (e.g., ["Barclays", "Barclays Capital"])
key_terms_table_headingThe exact heading text of the Key Terms table for this issuer. Used by html_extractor to find the right table.
table_labelsPer-issuer label → PRISM field path overrides (highest priority in label resolution)
section_headingsSection headings used to anchor the text window for LLM extraction
field_hintsPer-field synonyms, label aliases, and format notes for this issuer

Per-Field Instructions

Within a per-issuer hint, individual field_hints entries give Claude specific instructions for a single PRISM field for that issuer. For example, Barclays uses the label "Kick-In Barrier" for the barrier level — you can add that as a synonym for barrier.triggerDetails.triggerLevel.

Tab: Section Prompts

Section Prompts are used when Sectioned Extraction mode is enabled (see Extraction Settings). Instead of one large LLM call for all fields, the system makes multiple smaller calls, one per logical section of the PRISM schema.

The sections are:

  • identifiers — CUSIP, ISIN, product code
  • product — product name, denomination, notional, maturity date, currency
  • underlying_terms — underlying assets, initial/final levels
  • protection — barrier level, observation type, downside risk
  • autocall — call level, observation frequency, early redemption
  • coupon — coupon rate, barrier, frequency, payment schedule
  • parties — issuer, guarantor, calculation agent

Each section has a YAML config with a system prompt snippet and field list. You can edit these prompts in the Section Prompts tab. This is an advanced feature — the default single-call mode is suitable for most use cases.

Tab: Extraction Settings

This tab controls runtime extraction parameters. All settings are saved to files/runtime_settings.yaml and take effect immediately without a server restart.

SettingDefaultDescription
Claude Modelclaude-sonnet-4-6The Claude model used for all classify and extract calls. Choose from the registered models. Affects both cost and quality.
Sectioned ExtractionOffWhen enabled, extraction is split into 7 separate Claude calls (one per section). Slower but can improve results on long or complex filings.
Classification Gate Confidence0.80Filings with classification confidence below this threshold are blocked from extraction until manually confirmed. Prevents wasting budget on wrong product types.
Section Merge Delta0.15(Sectioned mode only) Minimum confidence improvement required for a section result to overwrite a value from a previous section.
Model pricing The model selector shows pricing for each available model. Claude Sonnet 4.6 is the default and provides the best balance of quality and cost for this use case. The Cost & Usage tab shows actual spend per model and can help you evaluate alternatives.

Tab: Label Map

The Label Map is a critical component of Tier 1 (HTML table) extraction. It maps label text from the Key Terms table (e.g., "Maturity Date:", "Downside Threshold:") to PRISM field paths (e.g., structuredProductsGeneric.maturityDate).

There are three label sources merged in priority order (higher = wins on conflict):

  1. Cross-issuer baseline (files/label_map_cross_issuer.yaml) — hand-curated, read-only in the UI
  2. User entries (files/label_map_user.yaml) — added via this tab
  3. Per-issuer overrides (table_labels in the issuer's YAML hints) — highest priority

Miss Log

The Miss Log panel (left side) shows labels that appeared in Key Terms tables but could not be resolved to any PRISM field path. For each miss:

  • Label — the raw label text as seen in the HTML
  • Count — how many times this label was seen across all filings
  • Sample value — an example value from the table (helps you understand what the field contains)
  • Issuer — the most recent issuer where this label appeared

Click the label to pre-fill it in the "Add Mapping" form. Then type the correct PRISM field path and click Add. This saves the mapping to label_map_user.yaml.

You can also click Dismiss to hide a miss from the log (e.g., for labels that you've decided not to map, such as footnote references or headers).

Mapping Panel

The right side shows all existing mappings. The baseline (cross-issuer) entries are shown in grey and cannot be removed here. User-added entries have a delete (×) button. You can also use this panel to add new mappings directly without going through the Miss Log.

When to add a mapping Add a mapping when you see a label in the Miss Log that clearly corresponds to a PRISM field, but the tool didn't know how to connect them. After adding, re-run extraction on affected filings to pick up the new mapping.

Tab: Schema Manager

The Schema Manager allows you to update the PRISM JSON schema without restarting the server. The PRISM schema (schemas/prism/prism-v1.schema.json) defines all payout types and their field structures. When PRISM evolves, you can fetch and review the new schema before activating it.

Checking for Updates

  1. Click Fetch Latest Schema. The backend fetches from the configured PRISM schema URL (http://10.10.21.57:30080/api/schema/json) and stores it in schemas/prism/pending/.
  2. The diff is computed automatically and displayed in the UI.

Reading the Diff

The diff classifies every change by severity:

SeverityMeaningAction Required
BreakingA field was removed or its type changed incompatibly. Existing extracted data may be invalid.Review all affected filings before activating.
CautionAn enum value was added or a field was made more permissive. Unlikely to break, but check.Review the specific change.
SafeA new field was added or a description changed. No impact on existing data.Safe to activate immediately.

Activating a New Schema

After reviewing the diff, click Activate to replace the active schema. The old schema is automatically archived to schemas/prism/archive/ with a timestamp. The schema loader reloads immediately — no server restart needed. All future classifications and extractions will use the new schema.

Tab: Log Viewer

The Log Viewer shows entries from the application log file (logs/app.log). Each log line shows the timestamp, level (INFO / WARNING / ERROR), module name, and message.

Filtering Logs

  • Level filter — show All, INFO+, WARNING+, or ERROR only

The Download Log button downloads the full app.log file as a plain text file. Useful for sharing with support or analysing long runs in a text editor.

6. Common Workflows

Workflow: Ingest and Extract a New Barclays Filing

  1. Click Filings in the top nav, then the Ingest tab in the sidebar.
  2. In the EDGAR Search box, type "Barclays" as the entity name and optionally a product keyword like "barrier reverse convertible". Set a date range if needed. Click Search EDGAR.
  3. Browse the results. When you find the target filing, note the accession number and click Ingest.
  4. The filing appears in the filings list with status ingested. Click it.
  5. Click Classify. Wait 5–15 seconds. The status changes to classified and the PRISM model is shown.
  6. If the model looks correct, click Extract. Wait 15–60 seconds.
  7. The field table appears. Review fields with AI badge and low confidence. Use Expert Review for side-by-side verification.
  8. Click Approve. Then Export to generate JSON + CSV.

Workflow: Why Are Some Fields Showing AI Badge Instead of TABLE?

The TABLE badge means the field was extracted from the Key Terms HTML table — deterministically, without AI. If a field shows AI instead, it means either:

  • The field was not present in the Key Terms table (it may be in a different section of the filing)
  • The label existed in the table but couldn't be mapped to a PRISM field path — check the Miss Log in the Label Map tab
  • The label was mapped but the parser returned None (value was in an unexpected format)

Steps to improve TABLE coverage:

  1. Go to Expert ⚙ → Label Map. Check the Miss Log for any labels from this filing's issuer.
  2. For each unmatched label that should map to a PRISM field, click the label, select the correct PRISM path, and click Add.
  3. Go back to the filing and click Re-Extract. The new mappings are picked up immediately.
  4. If the label was not in the Key Terms table at all, check the issuer's key_terms_table_heading in the Field Hints. If it's wrong, the extractor may be scanning the wrong table.

Workflow: A Field Value Is Wrong — How Do I Correct It?

  1. In the field table, find the incorrect field. Check its source badge — this tells you where the wrong value came from.
  2. Click the Edit icon on the field row.
  3. Read the source excerpt shown below the editor — this is the text Claude used. If the excerpt itself is wrong, the issue is in the filing or the hints.
  4. Type the correct value in the editor and click Correct. The field is saved with review_status = corrected.
  5. If the same error occurs repeatedly across filings from the same issuer, consider adding or updating a Field Hint (Expert ⚙ → Field Hints) to guide Claude to the right interpretation.

Workflow: A New Label Appeared in the Miss Log — How Do I Map It?

  1. Go to Expert ⚙ → Label Map.
  2. In the Miss Log panel, find the new label. Note the sample value — this tells you what the field contains.
  3. Look up the corresponding PRISM field path. You can refer to the active schema or the Tech Handbook's field reference.
  4. Click the label to pre-fill the mapping form.
  5. Type or paste the PRISM field path (e.g., coupon.rate) and click Add Mapping.
  6. Go to the affected filing(s) and click Re-Extract.

Workflow: The PRISM Schema Was Updated — What Do I Do?

  1. Go to Expert ⚙ → Schema.
  2. Click Fetch Latest Schema. The backend downloads the latest schema from the PRISM server.
  3. Read the diff carefully. Pay attention to any Breaking changes — these mean fields were removed or types changed.
  4. If there are breaking changes, assess the impact on existing approved filings before proceeding.
  5. If you are satisfied, click Activate. The old schema is archived and the new one is loaded immediately.
  6. For filings that were extracted with the old schema, consider re-extracting to populate any new fields.

7. Troubleshooting FAQ

The backend status shows "Backend down" (red dot)

The FastAPI server is not running or not reachable on port 8000. Check your terminal window running uvicorn. Common causes: the virtual environment is not activated, the server crashed (check logs/app.log), or another process is using port 8000.

"API key missing" warning in the header

The ANTHROPIC_API_KEY environment variable is not set. You cannot classify or extract without it. Set it before starting the backend: export ANTHROPIC_API_KEY="sk-ant-..."

Classification returned "unknown" or "needs review"

This happens when Claude's confidence was below 0.60 (forced to "unknown") or between 0.60 and 0.75 ("needs review"). Use Set Model to manually assign the correct PRISM payout type. The manual correction feeds back into future classification prompts as a few-shot example.

Extraction is blocked with "Classification confidence too low"

The classification confidence is below the gate threshold (default 0.80). This prevents spending extraction budget on filings that may be classified into the wrong model. Use Set Model to manually confirm the correct model, or lower the gate threshold in Extraction Settings.

Many fields show "Not found" after extraction

Possible causes: (1) The filing is classified into the wrong PRISM model — the schema expects different fields than what the filing actually contains. (2) The text window was not anchored on the Key Terms section — check if the issuer's section_headings are configured in Field Hints. (3) The filing is very long and the relevant section was beyond the 120K character limit — try enabling Sectioned Extraction mode.

HTML table extraction is not finding any fields (all AI badge)

The Key Terms table was not found in the HTML. Check: (1) Is the issuer's key_terms_table_heading set correctly in Field Hints? (2) Use Expert Review to open the filing HTML and verify the heading text of the Key Terms table. (3) The issuer may use a non-standard heading — add the correct heading to the issuer YAML hints.

Export button is greyed out

The filing must be in "approved" status before exporting. Click Approve first (all pending fields are auto-accepted on approval).

Ingest fails with "Filing not found on EDGAR"

The accession number may be wrong, or the filing may not be a 424B2 form. Verify the accession number on the SEC EDGAR website. Also check EDGAR rate limit logs — if the backend is making too many requests, EDGAR may temporarily block the IP.

Schema fetch fails with "502 Bad Gateway"

The PRISM schema server at 10.10.21.57:30080 is not reachable. Ensure you are connected to the LPA network or VPN. Check the Admin Log Viewer for the detailed error.

8. Glossary

424B2
The SEC form type used for prospectus supplements of structured products. Every new note issuance is accompanied by a 424B2 filing that describes the product's full economic and legal terms. All filings in this tool are 424B2 documents.
EDGAR
Electronic Data Gathering, Analysis, and Retrieval — the SEC's public filing system. All US-registered securities filings are submitted to and hosted by EDGAR. The base URL is https://www.sec.gov/cgi-bin/browse-edgar.
CUSIP
Committee on Uniform Securities Identification Procedures — a 9-character alphanumeric identifier assigned to each individual security. Every structured note has a unique CUSIP. The tool uses CUSIPs to look up the expected PRISM model from the CUSIP mapping spreadsheet.
PRISM
LPA's internal data model for structured financial products. PRISM defines a JSON schema with specific fields and types for each payout type (Barrier Reverse Convertible, Autocall Note, etc.). The goal of this tool is to populate PRISM-format JSON from raw 424B2 filings.
Payout Type / PRISM Model
A classification of structured products by their payoff structure. Examples: "barrierReverseConvertible", "autocallNote", "participationNote". Each model has its own set of PRISM fields. The classifier assigns one model per filing.
Tier 0 Extraction (Registry)
The highest-confidence extraction tier. Populates fields that are already known from EDGAR registry metadata — specifically CUSIP and issuer name. Confidence = 1.0. No document parsing required.
Tier 1 Extraction (HTML Table)
The second extraction tier. Parses the structured Key Terms HTML table present in most 424B2 filings, maps label text to PRISM field paths using the label map, and converts raw cell text to typed values using field parsers. Confidence = 0.97. Deterministic and fast.
Tier 2 Extraction (LLM)
The third extraction tier. Sends the filing text and PRISM schema to Claude AI via a structured tool call. Claude populates all schema fields it can find, with per-field confidence scores and source excerpts. Handles fields not covered by Tiers 0/1.
Accession Number
EDGAR's unique identifier for a filing submission. Format: XXXXXXXXXX-YY-ZZZZZZ (10-digit CIK, 2-digit year, 6-digit sequence). Used to locate the filing on EDGAR and as the local folder name (dashes removed).
Label Map
A configuration file mapping raw Key Terms table label text (e.g., "Maturity Date:") to PRISM dot-path field identifiers (e.g., structuredProductsGeneric.maturityDate). Three layers: cross-issuer baseline, user additions, and per-issuer overrides.
Miss Log
A database log of label strings that appeared in Key Terms tables but could not be resolved to any PRISM field path. Surfaced in the Expert Label Map tab to guide the user in adding new mappings.
Prompt Caching
An Anthropic API feature where the static portion of a prompt (system prompt, schema, hints) is cached on Anthropic's servers after the first call. Subsequent calls that reuse the same prefix only pay the cache-read rate (0.10× the regular input rate), reducing extraction cost by ~80–90% on repeated runs.
Confidence Score
A 0.0–1.0 score indicating certainty of an extracted value. Registry fields always score 1.0. HTML table fields score 0.97. LLM fields are self-reported by Claude: 1.0 = verbatim quote, 0.7 = inferred from context, 0.4 = estimated. Fields below 0.80 are highlighted in amber in the UI.
KPI Strip
The row of coloured metric chips shown below the filing header in the detail view. KPI = Key Performance Indicator. Shows timing, token counts, and USD cost for each pipeline step (Ingest, Classify, Extract).
Sectioned Extraction
An optional extraction mode where the schema is split into logical sections (identifiers, product terms, barrier, autocall, coupon, parties) and each section gets its own Claude call. More expensive but can improve results on complex or long filings.
