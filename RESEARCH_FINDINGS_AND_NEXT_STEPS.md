## 1. What Has Been Built

### 1.1 Pipeline Architecture (complete, working)

```
EDGAR Search API → Ingest (HTML download) → Classify (Claude, 2-stage)
                                                     ↓
                                           Extract (Claude, PRISM schema)
                                                     ↓
                                           Expert Review UI (FastAPI + React)
```

- **Two-stage classification**: Stage 1 uses 4,000-char cover page window (cheap, fast, ~80% coverage). Stage 2 extracts keyword-anchored sections (BARRIER, BUFFER, AUTOCALL, etc.) for ambiguous cases.
- **`unknown` model**: Explicit confidence floor (≥ 0.60). Below threshold → `unknown`, not a forced wrong match.
- **Post-extraction enum validation**: Every extracted value checked against schema enum constraints. Violations flagged as `schema_error` with `confidence = 0.0`.
- **Product name extraction**: Classifier returns `title_excerpt` (verbatim product title from page 1) and `product_features` ({type, features[], underlyings[]}) for every filing.
- **Feedback loop infrastructure**: `ClassificationFeedback` table stores corrections; `used_as_example` flag for future few-shot use.
- **Source excerpt traceability**: Every extracted field has `source_excerpt` (≤500 chars) linking the value back to filing text.

## 3. Motive AI POC — Key Insights (session 2026-01-08)

Robert Cox (Motive AI) presented a completed POC on the same extraction problem (PRISM schema from term sheets). Key differences and lessons:

### 3.1 Architecture Differences

| | Motive AI | Our Pipeline |
|---|---|---|
| LLM calls/doc | 8–10 (one per schema section) | 1 large call |
| Prompt hierarchy | General → section → **field-level rules** | Single system prompt |
| Section routing | Skip sections not applicable to product type | All fields always sent |
| Output enforcement | Anthropic tool/function calling | Raw JSON + regex cleanup |
| Feedback loop | Manual prompt iteration only | `ClassificationFeedback` table (ahead of them) |

### 3.2 Performance (unoptimised POC)
- Claude Sonnet: **~99% accuracy**, $1.51/document, 6 min/doc (architecture overhead)
- Claude Haiku: ~88% accuracy, ~$0.50/document
- 20 test documents (English + German), 25 runs with Sonnet

### 3.3 Most Valuable Insights for Our Pipeline

**Insight 1 — Field-level extraction rules are the biggest quality lever**
Their worst field was `coupon payment frequency` with a 3-word prompt ("monthly, quarterly, annually"). Fix: write a proper field-level instruction for every field. This directly explains our 40–55% fill rate — the schema alone doesn't tell Claude *where to look* or *how to interpret ambiguous presentations*.

→ **Action**: Add `extractionHint` per field (being generated by `files/issuer_extraction_hints.json` agent, see §5).

**Insight 2 — LLM-generate the initial field hints from sample filings**
Robert's team: "Give an LLM 20 term sheets and ask it to build the initial prompts. We use AI to generate those prompts and then AI to test performance."
→ **Action**: Feed Claude the 10-per-issuer local filings once ingested; ask it to generate field-level extraction rules per issuer. This is the `issuer_extraction_hints.json` workflow.

**Insight 3 — Section-by-section extraction beats one big call**
8–10 calls per document, one per schema section. Benefits: smaller context → less malformed JSON; section-specific prompts; skip non-applicable sections (e.g. no autocall section for a plain barrier product).

→ **Future action**: For filings > 80K chars stripped text, split extraction by section group. Not urgent now; add to P2 backlog.

**Insight 4 — Pre-filter document pages before extraction**
Robert: "A preliminary extraction step classifies pages to relevant areas." Markus: "A typical term sheet has 5–6 sections, only 3 are relevant."
→ Our stage-2 keyword window already does this for *classification*. Apply same logic to *extraction input*: identify key terms section (typically first 200 lines for BMO, lines 147–280 for JPMorgan) and send only that section.

**Insight 5 — Three error types, ranked by severity**
Christian (LPA): "(1) null when value exists: unfortunate but not terrible. (2) value extracted where field doesn't apply: a major problem. (3) wrong value: worst."
→ Validates our `not_found` vs `schema_error` distinction. Expert review UI must make type (2) and (3) errors visually prominent.

**Insight 6 — Classification review gate before extraction**
Their workflow: classify → explicit human review → only then run 8-10 extraction calls. Prevents wasting extraction budget on a wrong product type.
→ **Action**: Add `CLASSIFICATION_GATE_CONFIDENCE = 0.80` — below this, set `status = needs_classification_review` and block extraction until confirmed.

**Insight 7 — Reprocessing queue for layout changes**
When an issuer redesigns their term sheets, field fill rate drops suddenly. Their design: flag documents → hold → update prompts → trigger batch rerun.
→ Our `status` field already supports this (reset to `classified`, re-run extraction). Missing: automated detection that quality dropped for a given issuer.

**Insight 8 — PDF input: OCR to text, store source with coordinates**
They use Tesseract OCR (PDF → plain text) before sending to LLM. Store source PDF coordinates per extracted field for auditability.
→ Confirms `pdfplumber` / Tesseract is the right path for our PDF support (P4 item). No change to extraction LLM layer needed.
